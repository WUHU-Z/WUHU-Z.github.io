1.什么是特征工程？常见的特征工程方法有哪些？

定义：特征工程就是把原始数据变成能让机器学习模型更好“理解”和“利用”的特征的过程，常见方法有特征选择、特征提取、特征构造和特征变换。

为什么重要？
好的特征能让模型更容易发现数据中的规律，提升预测准确率。特征工程常被称为“数据科学的核心竞争力”。

常见的特征工程方法

特征选择（Feature Selection）

定义：从众多特征中挑选出最有用的部分，去掉无关或冗余的特征。

特征提取（Feature Extraction）

定义：将原始特征转换成新的、更有代表性的特征。

特征构造（Feature Construction）

定义：根据已有特征，通过计算、组合等方式创造新特征。

2.部署一个模型以及推理一个模型分别用到的参数量怎么估算

  • 部署一个模型所需的参数量等于模型本身的总参数量（即所有可训练参数的浮点数个数），通常以float32存储时为4字节/参数；

  • 而推理时实际使用的参数量在数值上与部署参数量相同，但有效参与计算的参数量可能因优化技术（如剪枝、量化、KV Cache复用、层跳过等）而显著减少；

  • 更关键的是，推理阶段的内存占用不仅包含参数，还包括激活值（activations）、KV缓存（对Transformer类模型）、临时缓冲区等，其总量往往远超参数本身。因此，“部署参数量”是静态的、确定的；“推理参数量”在严格意义上仍是全部参数，但推理系统关注的是端到端内存带宽消耗和计算访存比，需综合参数量、batch size、sequence length、精度、缓存机制等联合估算。

3.模型prefill和decoding的区别了解吗？KV cache的大小是多少，怎么算

Prefill（预填充）阶段是指大语言模型在生成响应前，对输入的完整提示词（prompt）进行一次前向传播，逐层计算所有token的隐藏状态，并将每层Transformer中自注意力机制产生的Key和Value向量缓存到KV Cache中；Decoding（解码）阶段是指模型基于prefill结果，以自回归方式逐个生成新token：每次仅输入一个token（即上一轮输出的token），复用prefill已缓存的KV，并为当前新token计算新的K/V，追加到对应层的KV Cache末尾。

KV Cache的大小不是固定值，而是动态可变的，其总显存占用 ≈ 2 × num_layers × batch_size × seq_len × hidden_size × sizeof(dtype)

4.了解Deepspeed的原理吗？假设对矩阵A×B这个运算，给4张卡，简述是如何分配矩阵A，B参数并在显卡中进行交互来达到节省显存的效果的

DeepSpeed 通过 ZeRO（Zero Redundancy Optimizer）技术实现显存优化，在矩阵乘法 A×B 的场景下，若使用 4 张 GPU 进行分布式前向/反向计算，典型做法是结合 张量并行（Tensor Parallelism） 和 ZeRO-3 阶段的参数分片（Parameter Sharding） 来分配 A 和 B，并通过跨卡通信（如 all-gather / reduce-scatter）完成局部计算与结果聚合，从而避免每张卡存储完整 A、B，显著节省显存。

  • 张量并行（Tensor Parallelism）：负责“怎么算”

  • ZeRO-3（参数分片）：负责“谁存什么”

MatMul优化

  • refill阶段的Matmul优化主要聚焦于大尺寸、一次性计算的KV缓存构建，适用批处理GEMM（如cuBLAS GEMM、FlashAttention-2的分块预填充）、内存布局重排（BNSD → BNHS）、FP16/BF16混合精度+Tensor Core加速，并支持动态batch size与变长序列；
  • Decoding阶段的Matmul优化则面向低延迟、高吞吐的单token自回归生成，核心是PagedAttention（KV缓存分页管理）、Key-Value Cache复用、TinyGEMM（小矩阵乘法定制内核，如1×d_head × d_head×d_model）、RoPE位置编码融合、以及基于prefetch + register tiling的极致访存优化（如vLLM的block-wise attention kernel）。

5.Temperature,Top-k,Top-p

  • Temperature 是一个正实数缩放因子，作用于 logits（未归一化的原始输出分数），数学上是对 softmax 输入进行缩放：$P_i = exp(logit_i / T) / Σ_j exp(logit_j / T)$。当 T → 0 时趋向贪婪解码（确定性最高）；T = 1 为标准 softmax；T > 1 使分布更平滑、尾部概率增大，输出更随机/创造性。
  • Top-k 采样：仅保留 logits 中数值最大的 k 个词元，其余置为 -∞，再在该子集上做 softmax。k=1 等价于贪婪解码；k 较大（如 50）保留更多候选，增强多样性。
  • Top-p（Nucleus Sampling）：按概率从高到低累加，截断至累积概率首次 ≥ p 的最小前缀集合，仅在此集合内重归一化采样。p=1.0 等价于全词表 softmax；p=0.9 表示只考虑覆盖 90% 概率质量的最小词元子集，动态适应分布稀疏性（如长尾分布时自动选更多词，尖峰分布时只选少数）。

6.softmax相关

  • 为什么需要Softmax？——因为线性分类器（如全连接层）输出的是未归一化的logits（实数），不能直接作为概率；而Sigmoid仅适用于二分类，多分类需满足“互斥且完备”约束，即概率和为1且非负，Softmax天然满足；

    • sigmod每个输出是独立的（回答是否）–>既像猫又像狗；
    • softmax每个输出相互竞争(回答更像哪个)–>像猫就不能像狗

  • 为什么用exp？——exp(x) 单调递增、处处可导、将实数域映射到正实数域，是构造正权重最自然的选择；

  • 为什么必须做减法平移（如减去max(z)）？——防止exp(zᵢ)上溢出（overflow），这是实际部署中的关键工程细节

7.通信算子

通信算子（Communication Operators）是分布式训练框架（如PyTorch Distributed、TensorFlow Collective Ops、Horovod、DeepSpeed）中用于在多个计算设备（GPU/CPU节点）之间同步或交换张量数据的一组底层原语。

常见通信算子包括AllReduce、Broadcast、AllGather、ReduceScatter、Send/Recv、Barrier等，它们直接映射到MPI或NCCL等高性能通信库的接口，是实现数据并行、模型并行、流水线并行等分布式训练策略的基础设施。

9.MCP Server

MCP Server 是一个基于 JSON-RPC 的工具服务框架，它将外部能力（API、数据库、RAG 等）抽象为标准化 tools，通过 tools/list 和 tools/call 接口供 LLM 调用。

其核心结构包括协议层、工具层和资源层，本质是为大模型提供一个安全、可扩展、可组合的外部能力执行环境，使模型从“纯生成”扩展为“可调用工具的智能体”。

让 LLM 能以统一协议访问工具、数据源和业务能力。LLM = 决策器，MCP Server = 执行器

架构

一个标准 MCP Server 通常分三层：

  • 协议层（Protocol Layer）：负责和 LLM / Host 对话
  • 能力层（Tool Layer）：调用工作负责干活
  • 资源层（Resource Layer）:提供上下文数据

流程

  1. 用户输入问题进入大模型
  2. 模型判断是否需要调用工具，并生成结构化 Tool Call（包含 tool_name 和 parameters）
  3. Agent Runtime 校验并解析该调用请求
  4. Runtime 将请求发送到 MCP Server 执行对应工具
  5. MCP Server 执行后返回标准化结果（success/error + output）
  6. Runtime 将结果回填到模型上下文
  7. 模型基于工具结果生成最终回答或继续调用下一步工具

10.工具多导致token数过多的解决办法

解决思路是：不把所有工具都塞给模型，而是按需选择和压缩工具信息。

核心方法有四类：

  1. 动态加载工具：运行时按需注入，而不是全量放进 prompt
  2. 工具摘要化：只保留工具名称 + 简短功能描述 + 参数结构
  3. 工具检索/筛选（Top-K）：根据用户 query 只选最相关的少量工具
  4. 分层调用（Router + Main Model）：先用轻量模型或规则做工具路由，再让大模型执行具体调用
  5. 结果压缩：工具返回后做摘要，只把关键字段回传给模型

11.昇腾NPU架构对Transformer友好吗

昇腾NPU（如Ascend 910系列）架构在硬件设计和软件栈层面均对Transformer类模型具有高度友好性，主要体现在其原生支持稀疏计算、大带宽HBM内存、定制化矩阵运算单元（Cube Unit）、高并发张量调度能力，以及CANN（Compute Architecture for Neural Networks）软件栈对Attention、LayerNorm、GeLU等Transformer核心算子的深度优化。

尤其通过AI Core中的Cube引擎可高效执行FP16/BF16混合精度GEMM，配合AI CPU与DVPP协同完成Tokenizer/De-tokenizer与数据预处理，形成端到端Transformer加速闭环。

12.一个量化后的大模型（如INT8/4）运行时内存占用大概多少

一个量化后的大模型（如INT8或INT4）的运行时内存占用，主要由模型参数存储、激活值（activations）、KV缓存（推理时）、优化器状态（训练时）以及框架开销共同决定。

以纯参数存储为例：若原始FP16模型参数量为N（单位：参数个数），则INT8量化后参数内存 ≈ N × 1 字节，INT4量化后 ≈ N × 0.5 字节；但实际运行时内存远高于此——典型场景下，推理时总内存 ≈ 参数内存 + KV缓存内存 + 激活内存（含中间张量）+ 对齐/填充/框架元数据开销。

例如，Llama-3-8B（约8.05B参数）INT4量化后参数仅占约4.0 GB，但使用FlashAttention-2 + PagedAttention（vLLM）在batch_size=1、max_seq_len=2048时，实测GPU显存占用通常为4.8–5.5 GB；若开启prefill+decode混合调度、多batch或长上下文（如32k），KV缓存可暴涨至参数内存的1–3倍。
