1.HTTP和HTTPS的区别

总结：HTTP 像明信片——谁都能偷看；HTTPS 像贴好身份证、加密又防篡改的快递箱，既保证内容机密，也验证“寄件人”身份。

维度 HTTP HTTPS
端口默认 80 443
传输层 直接跑在 TCP 之上 先经 TLS（SSL）再走 TCP；TLS 1.3 已与 QUIC 融合到 HTTP/3
加密 ❌ 明文 ✅ 对称加密（AES/ChaCha20）
完整性校验 ❌ 无 ✅ MAC / AEAD，防篡改
身份认证 ❌ 无 ✅ 服务器证书（可选客户端证书）
防御能力 易遭窃听、伪造、劫持（MITM） 防窃听、防篡改、防冒充，但不解决业务逻辑漏洞

HTTPS的握手过程

HTTPS 的握手过程，本质上是先建立 TCP 连接，然后进行 TLS 握手来协商加密参数、验证服务器身份并生成会话密钥。

2.传输协议怎么选择？TCP 和 UDP 区别，UDP 优势是什么，TCP 如何建立连接，若使用 UDP ，如何在应用层支持可靠性

场景：

  • TCP 适用于要求数据完整、有序、不丢失的场景（如HTTP、FTP、SMTP）；
  • UDP 适用于对延迟敏感、可容忍少量丢包、需低开销或支持多播/广播的场景

优势：

UDP 的核心优势是无连接、无握手、无重传、无拥塞控制、头部开销小（仅8字节）、支持一对一/一对多/多对一通信。

连接：

TCP 通过三次握手建立连接：客户端发送SYN，服务器回复SYN+ACK，客户端再发ACK，双方进入ESTABLISHED状态。

三次握手的核心目标是双向通信能力的可靠确认。第一次握手让服务器知道客户端“能发”；第二次握手让客户端知道服务器“能收也能发”（因为服务器在响应中既发了SYN又发了ACK）；但此时服务器还不知道客户端是否“能收”

3.应用层方面协议怎么设计，交换什么信息

应用层协议设计的核心是定义通信双方在特定业务场景下“说什么、怎么说、何时说、如何确认”，即需明确消息语义（交换什么信息）、语法结构（数据格式与编码）、时序规则（交互流程与状态机）、错误处理与可靠性保障机制

物链网输会示用

4.Redis和数据库的关系

  1. 数据库 (MySQL等)**：作为主存储**。它基于磁盘，容量大、成本低，能保证数据的持久化和强一致性，支持复杂的事务和查询。它是数据的“唯一真理来源”。
  2. Redis：作为缓存层。它基于内存，速度极快，但容量有限、成本较高。它存储的是从数据库中读取的“热点数据”副本，用于加速访问。

类似计组中cache与内存的关系，通过写直达+非写分配（防止污染cache）/写回+写分配

5.Redis的数据类型

Redis核心数据类型包括：

  1. String：存储文本、数字或二进制数据。
  2. List：双向链表，支持队列和栈操作。
  3. Hash：字段-值映射，适合存储对象。
  4. Set：无序唯一集合，支持交并差运算。
  5. Sorted Set：按分数排序的有序集合。扩展类型含Bitmaps、HyperLogLog、地理空间及Stream（消息队列）。

6.Redis分布式锁

Redis分布式锁是一种在分布式系统中协调多个服务节点对共享资源进行互斥访问的机制，典型应用场景包括库存扣减、订单幂等处理、定时任务防重复执行、秒杀场景下的并发控制等。

可以用redis锁实现多进程的同步

服务端如何保持原子性：Redis 服务端通过单线程事件循环（基于 Reactor 模式）+ 命令的串行执行 + 内置数据结构的原子操作实现命令级原子性；对多命令原子性需求（如多个 key 操作），则依赖 Lua 脚本或事务（MULTI/EXEC）配合 WATCH 机制实现乐观锁语义下的条件原子性。

7.确保Redis与数据库双写一致性

常用方法包括：先更新数据库再删缓存（Cache-Aside）、延迟双删（更新前后均删缓存并延迟二次删除）、基于消息队列异步同步、监听数据库，触发缓存更新，以及设置缓存过期时间兜底。需结合业务场景选择策略，配合重试机制确保最终一致性。

策略 核心思想 适用场景 一致性级别
Cache Aside 读时优先读缓存，写时先更新数据库再删除缓存 通用场景（读多写少） 最终一致性
延迟双删 先删缓存，更新数据库后，延迟一段时间再次删除缓存 高并发读写、热点数据 最终一致性
消息队列异步同步 通过消息队列异步更新缓存，保证最终一致性 高吞吐、允许短暂延迟 最终一致性
分布式锁 写操作时加锁，确保同一数据串行更新 强一致性要求的高并发场景 强一致性
TCC 模式 业务层分阶段提交（Try-Confirm-Cancel），保证数据库和缓存操作的原子性 金融级强一致性场景 强一致性

8.聚簇索引和非聚簇索引

特性 聚簇索引 (Clustered Index) 非聚簇索引 (Non-Clustered / Secondary Index)
本质 索引即数据，数据与索引存储在一起 索引与数据分离，索引只存主键值
叶子节点内容 存储完整的行数据 存储索引列的值 + 主键值 (ID)
数量限制 每张表只能有 1 个 每张表可以有多个
查询效率 极快（一次查找即可获取数据） 需回表（先查索引得ID，再查聚簇索引得数据）
物理存储 决定了数据在磁盘上的物理排列顺序 逻辑顺序，与数据物理存储顺序无关
适用场景 主键查询、范围查询、排序 辅助字段查询、覆盖索引查询

Redis Hash 是一种“数据结构”（用来存数据的），而非聚簇索引是一种“查询加速机制”（用来查数据的）。

一个本身就是数据，一个依附于数据

9.可重复读解决了什么问题

可重复读位于读已提交之上、串行化之下。

其核心诉求是：同一个事务内多次执行相同的SELECT语句，应看到完全一致的结果集（即不会因其他事务的并发修改而改变）。

可重复读（Repeatable Read）隔离级别主要解决了脏读和不可重复读问题，但不解决幻读问题；在MySQL的InnoDB引擎中，通过多版本并发控制（MVCC）配合间隙锁（Gap Lock）或临键锁（Next-Key Lock），实际上在大多数场景下也避免了幻读，从而实现了“严格可重复读”。

脏读（读到未提交的修改）–>读到的是“草稿”，随时可能被回滚（作废）。

不可重复读（同一行数据两次读取值不同，因被其他事务update/commit）–>读到的是“已提交的真相”，但破坏了事务的一致性假设。

“幻读”在SQL标准中特指：同一事务中，第一次SELECT得到N行，第二次相同SELECT得到N+1行（因其他事务INSERT了一行并提交）–>读到的是“凭空出现/消失的行”，导致范围判断失效。

10.MVCC

核心思想：为数据行维护多个历史版本，让读操作和写操作可以并行执行，互不阻塞，从而大幅提升数据库的并发性能。

核心价值

  • 读写不阻塞：读事务读取旧版本数据，写事务创建新版本数据，两者互不干扰，解决了传统锁机制中“读阻塞写、写阻塞读”的性能瓶颈。
  • 实现事务隔离：它是实现 READ COMMITTED（读已提交）和 REPEATABLE READ（可重复读）这两种关键隔离级别的基础，有效避免了脏读和不可重复读问题。

11.间隙锁是什么，发挥什么作用

间隙锁（Gap Lock）是MySQL InnoDB存储引擎中的一种行级锁类型，用于锁定索引记录之间的“间隙”（即不存在实际记录的索引区间），而非具体某一行。

例如，在索引值为(1, 5, 10)的B+树中，间隙包括(-∞,1)、(1,5)、(5,10)、(10,+∞)，间隙锁可锁定其中任意一个开区间。

12.AOP 和 IOC

IOC（控制反转） 是一种设计思想，它把对象的创建和管理权从代码中剥离，交给 Spring 容器，从而实现解耦；

AOP（面向切面编程） 是一种编程范式，它把日志、事务等通用的‘横切逻辑’从业务代码中抽取出来，实现复用

13.进程线程

  • 进程是操作系统进行资源分配和调度的基本单位，拥有独立的内存空间（包括代码段、数据段、堆、栈等）、文件描述符、信号处理机制等系统资源；
  • 线程是CPU调度的基本单位，是进程内的一个执行流，同一进程内的多个线程共享该进程的虚拟地址空间（如全局变量、堆、代码段、已打开文件等），但每个线程拥有独立的栈空间、寄存器状态（如PC、SP）、线程ID、信号掩码和errno等私有资源。

进程 —— 相当于“工厂”或“容器”

*线程 —— 相当于“工人”或“流水线”**

线程是组成进程的实体。一个进程可以包含多个线程，这些线程是并行或并发执行的，而不是像微指令那样按顺序拆解执行。

13.死锁条件和解除方法

死锁发生的四个必要条件是：互斥条件、占有并等待条件、非抢占条件（不可剥夺条件）、循环等待条件。

解除死锁的常用方法包括：终止进程（剥夺资源或回滚）、资源抢占（强制回收部分进程已占有的资源）、进程回退（基于检查点的撤销与恢复）、死锁检测与恢复（周期性检测+选择牺牲进程）

13.僵尸进程、孤儿进程

  • 僵尸进程：子进程先死（调用exit），父进程不收尸（没有接收子进程退出状态）。
    • 占用进程号（PID）和进程控制块（PCB），大量僵尸进程会耗尽系统的 PID 资源，导致系统无法创建新进程
    • 不能直接用 kill -9 杀死僵尸进程，因为它已经“死”了。
    • 解决方法：杀死它的父进程，或者让父进程调用 wait() 回收。
  • 孤儿进程：父进程先死，子进程被领养（通常无害）。
    • 通常无害
    • 会被系统中的 1 号进程自动“领养”
    • 当孤儿进程最终结束时，init 进程会负责调用 wait() 回收它的资源，防止它变成僵尸进程

14.Kill -9 （SIGKILL 不能被捕获和忽略）若无法杀死另一个进程为什么（无权限）

  • 权限不足
  • 僵尸进程
  • 不可中断睡眠（状态显示为D,底层问题难以解决–>重启系统）

15.数据结构的底层

  1. vector:动态数组（支持随机访问，不适合中间插入删除）
  2. list：双向链表（不支持随机访问，适合中间插入删除）
  3. deque：分段连续的动态数组（维护中控器（Map），这是一个指针数组，每个指针指向一块固定大小的连续缓冲区）
    • queue和stack：作为容器适配器，默认使用deque，也可以使用vector和list
  4. map / set / multimap / multiset
    • 底层结构：红黑树（Red-Black Tree)是一种自平衡二叉搜索树。它通过复杂的旋转操作（左旋、右旋）和颜色变换，保证树的高度始终维持在 **O(log n)**。
    • 优点：查找、插入、删除的时间复杂度稳定在 **O(log n)**；支持范围查询（如 lower_bound）；数据有序。
    • 缺点：节点需要存储颜色、左右子节点指针，内存开销大；树节点的内存不连续，缓存不友好。
  5. unordered_map / unordered_set
    • 底层结构：哈希表（Hash Table）
    • 优点：查找、插入、删除的平均时间复杂度为 **O(1)**，速度极快。
    • 缺点：数据无序；哈希函数计算有开销；最坏情况下（哈希冲突严重）性能退化为 O(n)。
  6. priority_queue
    • 和queue，stack一样是容器适配器，底层是vector，也可选deque
    • 通过封装底层的容器，并配合堆算法（push_heap, pop_heap 等），对外提供了一个“永远只让优先级最高的元素出头”的接口。
