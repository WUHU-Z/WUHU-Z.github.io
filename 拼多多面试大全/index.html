1.训练不稳定如何排查

  • loss上下乱跳 → 学习率/数据问题

    • 学习率过大（波动大）/过小,收敛慢
      • warmup或者scheduler
    • 有没有 异常值 / 极端值，label 对不对
      • 通过数据标准化，数据清洗，数据增强，batch_size调整解决

  • 梯度爆炸

    • gradient clipping

    • 用更稳定的优化器（Adam > SGD）

    • 检查 loss 计算有没有 log(0) 这种作死操作

    • 权重初始化过大

    • 网络太深

  • 梯度消失

    • 激活函数（sigmod）导致梯度消失
    • 权重初始化过小
    • 网络太深

  • 训练能收敛但效果差（过拟合）

    • dropout

    • 权重衰减

    • L2正则化

2.Long Context 常见思路有哪些？在业务里如何做”能看长文本但不太贵”的折中?

技术上

  • 使用原生长长上下文模型去进行推理（通过改进位置编码和注意力，使模型能记住更远的信息）

  • 检索增强生成 (RAG, Retrieval-Augmented Generation)

    • 将长文档切分为小块（Chunks），向量化存入数据库。用户提问时，只检索最相关的几个 Chunk 放入 Prompt。

  • 混合架构与优化技术 (Hybrid & Optimization)

    • 滑动窗口注意力
    • KVcache压缩（对 KV Cache 进行低精度量化）
    • 层次化摘要：对文档局部进行摘要，再对摘要进行摘要，形成树状结构。推理时自顶向下展开相关分支

业务上

  • 动态路由：用一个极小的 Classifier 模型（或规则）判断用户意图，分发到不同管道。简单任务用小模型，复杂任务用long context模型

  • **RAG + 长窗口” 混合模式 (The Sweet Spot)**：先用RAG检索topK个chunks，把对应chunks的上下文传入

  • 分层摘要与“地图 - 导航”机制：推理时传入全局摘要和局部关键原文

3.电商领域预训练数据：低质过滤与去重会怎么做？用哪些简单指标快速判断数据质量？

1.过滤

  • 首先进行基础规则过滤，将长度过短，目标语言占比过低，含有特殊符号和乱码，敏感词和广告的数据进行过滤
  • 启发式指标过滤，使用通用小语言模型（qwen、LLama），将困惑度 (Perplexity, PPL) 较小的文本筛去
  • 模型判别过滤，训练二分类模型，过滤低质量或包含个人信息的数据

2.去重

  • Hash去重，计算文档的hash值，通过hash算法进行去重
  • embeding去重，通过向量模型将文本向量化，然后计算余弦相似度
  • 基于 SPU 的去重（电商常用）数据源包含结构化元数据（商品 ID、品牌、型号），优先基于 SPU (Standard Product Unit) 进行聚合，对于同一 SPU 下的多条文本描述，保留信息熵最高（字数适中、形容词丰富、包含具体参数）的一条。

4.多模态模型的结构

图像编码器 → 对齐/连接层 → LLM

4.1 图像编码器

把图片变成 embedding

CNN（老派但稳），ViT（主流），CLIP encoder（最常见）

4.2 连接层

  • 线性投影/Adapter
    • 表达能力有限
  • 交叉注意力
    • 显存消耗过大
  • Q-Former
    • 通过可学习的query去提取图像信息

4.3 LLM

把“视觉+文本”统一成语言输出

常见坑

1.模态对齐失败

  • 表现：输出和图像毫无关系
    • projector 太弱
    • embedding 空间没对齐
    • 训练数据不够好

2.图像信息丢失

  • 表现：输出和图像毫无关系
    • token 太少
    • pooling 太 aggressive

3.token 数量爆炸

  • 表现：显存爆炸（OOM）
    • 图像patch太多

如何检查图文一致性／不编造信息？优先加哪些自动化检查？

  • 实体存在性验证

    • 提取模型回答中的名词实体、属性、数值、空间坐标，通过轻量级监测模型在原图中搜索，进行置信度比对

  • 多模型投票

    • 用两个不同架构的模型对同一张图生成描述，查看是否有冲突

  • 基于“不确定性”的自检

    • 若关键名词logits过低，触发“拒绝回答”机制，或让模型重新审视图片

  • 思维链验证

    • 强制模型在输出最终结论前，先输出“证据链”，若链出问题则不用看结果

5.SFT 数据如何更贴近业务：如何避免过度模板化？如何做基础的 train / test 去重来避免评测失真？

核心：打破数据的工业化痕迹，并建立严格的数据隔离墙。

  1. 避免过度模版化
    • 为不同任务赋予不同人设，让模型学会根据语境调整语气
    • 句式重组，利用大模型为每条核心业务数据至少生成 3-5 种不同的提问方式
    • 在构造数据时，随机移除显式的指令词，让模型学习从上下文推断意图（无需请总结…）
  2. 内容动态化（噪声注入）
    • 引入拟声词或错别字，模拟真实场景
    • 多轮对话逻辑构建
    • 文本长度随机化
  3. 业务决策痕迹
    • 直接从日志，用户反馈中提取训练数据
    • 保留用户的原始措辞（包括不规范的表达），只由人工或大模型修正“助手”的回答部分

6.对齐（ RLHF / DPO 等）整体思路是什么：为什么需要偏好对／奖励信号？

  • 预训练 (Pre-training) 让模型学会了“世界是怎么样的”（知识、语法、逻辑）。
  • SFT (监督微调) 让模型学会了“怎么回答问题”（遵循指令的格式）。
  • 对齐 (RLHF/DPO) 让模型学会“怎么回答得更好“（更诚实、更安全、更符合人类偏好、更有用）。

  1. RLHF (Reinforcement Learning from Human Feedback)

    • 三步走：收集偏好数据 -> 训练奖励模型 (Reward Model, RM) -> 强化学习 (PPO) 优化策略。
    • 特点：理论最完善，效果上限高，但工程极其复杂，训练不稳定，显存消耗巨大。

  2. DPO (Direct Preference Optimization)

    • 一步走：数学推导证明，可以直接在偏好数据上优化策略模型，不需要显式训练奖励模型，也不需要复杂的 PPO 采样过程。

    • 特点：稳定、高效、省显存，目前业界主流。
      $$
      \mathcal{L}_{DPO}
      = -\mathbb{E}\left[
      \log \sigma \left(

      \beta \log \frac{\pi_\theta(y_w \mid x)}{\pi_{\text{ref}}(y_w \mid x)}

      \beta \log \frac{\pi_\theta(y_l \mid x)}{\pi_{\text{ref}}(y_l \mid x)}
      \right)
      \right]
      $$

给定一个提示词（Prompt）和两个回答（一个好Chosen，一个坏Rejected），调整模型参数，使得模型生成“好回答”的概率相对于“坏回答”的概率最大化

​ 贝塔用于控制对参考模型的约束程度，决定了模型在多大程度上可以偏离参考模型。

​ sigma把损失值压缩到0~1表示选出正确偏好对的概率

  1. 其他变体：KTO (基于 Kahneman-Tversky 效用理论，只需单条数据的标签而非成对比较)、ORPO (在 SFT 阶段直接融入偏好优化) 等。

为什么需要“奖励信号” (Reward Signal)

  • 突破 NLL 的局部最优

    • 通过强化学习（或 DPO 的隐式梯度）反向传播，强行调整模型参数，使其向“高分区域”移动，哪怕这意味着要降低某些局部 token 的概率。突破数据的约束。

  • 处理“信用分配” (Credit Assignment) 问题

    • SFT 很难定位是哪几个 token 的问题（例如泄露了隐私，或者逻辑转折错误），它只能全盘否定或肯定参考文本。
    • 奖励模型（尤其是经过训练的 RM）能够识别出导致低分的关键特征，并通过 RL 算法（如 PPO 的优势函数）告诉策略模型：“在这个状态下，采取这类动作会导致长期回报下降”，从而更精准地修正行为

  • 统一多维度的优化目标

    • 人类偏好是多维度的：**有用性 (Helpfulness) + 诚实性 (Honesty) + 无害性 (Harmlessness)**。
    • 奖励信号可以将这三个维度加权融合成一个单一的标量值。模型只需要优化这一个值，就能自动在多个目标之间找到最佳平衡点（Pareto Optimal），而不需要人工设计复杂的规则权重。

7. Prompt ／模板如何管理：如何版本化、如何回滚、如何避免一次改动导致整体波动？

核心原则：Prompt即代码

不要将 Prompt 硬编码在 Python/Java 代码字符串中，也不要散落在 Excel 或 Wiki 里。

  • 独立存储：将 Prompt 提取为独立的配置文件（JSON/YAML/Markdown）。
  • 结构化元数据：每个 Prompt 必须包含版本号、作者、适用模型、创建时间、测试用例 ID 等元数据。
  • **单一事实来源 (SSOT)**：线上运行的 Prompt 必须直接读取自版本控制系统，禁止人工在后台手动修改。

回滚一般要做到两个层面：配置级回滚和流量级回滚。

  • 配置级回滚就是直接切回上一版模板；
  • 流量级回滚就是只回滚某个场景或某个实验桶，而不是全量切换。

​ 为了避免一次改动导致全局波动，通常会先小流量灰度，再看关键指标，比如回答正确率、拒答率、转人工率、异常率和 token 成本，没有问题再逐步扩大流量。

灰度：新旧版本“同时存在”

8.多语言场景怎么评估？如何做一套简单可执行的离线评测集？如何覆盖不同语言与类目

​ 多语言评测不能只把中文题翻译一下就结束，必须覆盖不同语言真实表达习惯。做离线评测集时，先按语言拆，比如中文、英文、东南亚语种，再按业务类目拆，比如商品问答、规则说明、客服场景、售后场景、活动场景。每个语言和类目下都要覆盖高频问题、长尾问题、歧义问题、拒答问题和多轮问题。

​ 一套简单可执行的离线评测集可以先按“语言 × 类目 × 场景类型”做分层抽样，每一层挑一批真实 query，再标注标准答案或证据范围。评测时不只看答案文本是否完全一致，更看是否答对、是否引用了正确证据、是否遵守语言要求。多语言场景还要额外看语言漂移，比如用户用泰语提问，系统不要夹杂过多中文或英文。

9.A/B 实验怎么落地？分桶、周期、显著性怎么处理？遇到大促波动时如何解释实验结果？

​ A/B 实验怎么落地？分桶、周期、显著性怎么处理？遇到大促波动时如何解释实验结果？ A/B 实验首先要保证分桶稳定，不能用户今天在 A，明天在 B。一般按 user_id 或 device_id 做一致性哈希分桶，这样同一个用户在实验周期内始终落在同一个桶。周期上至少要覆盖业务波动的完整周期，比如工作日和周末，不然很容易被时段效应误导。 显著性通常看核心指标的样本量、均值差和置信区间，不能只看单日涨跌。

大促场景

​ 大促期间直接看整体指标（Aggregate Metrics）往往是失效的。因为大促会引入两个巨大的干扰变量：用户结构变化（大量羊毛党/低频用户涌入）和Query 分布漂移（咨询问题从“日常用法”突变为“发货/退款/优惠”）。

​ 大促期间的波动要单独分析，因为用户结构、问题分布、系统负载都会变，直接把大促期和普通期混着看很容易误判。更稳的做法是分层看：新老用户分层、活动与非活动分层、意图分层（将用户 Query 分为“售前咨询”、“售后维权”、“活动规则”）

10.RAG 流程是什么

RAG 一般分离线和在线两部分。

  • 离线侧先把文档接入，做清洗、去重、切片、embedding，然后写入向量库或混合检索库。
  • 在线侧用户问题进来后，先做 query 改写或者纠错，再做检索召回，取回候选片段后做 rerank，最后把问题和证据一起送给大模型生成答案。

如果是 Agent 场景，RAG 不是独立模块，而是和工具调用、记忆、状态管理一起协同的。比如用户问规则类问题就直接走检索，问实时数据就调工具，问混合问题就先查知识再结合工具结果生成答案。

11.多 Agent 协作机制怎么做

​ 多 Agent 协作一般不是让每个 Agent 随机聊天，而是做职责拆分。最常见的是一个 Planner 或 Router 负责拆解任务，再交给不同的执行 Agent。比如检索 Agent 负责查知识库，Tool Agent 负责调业务接口，Summary Agent 负责总结结果，Judge Agent 负责判断结果是否可用。

​ 这样拆的好处是每个 Agent 的 prompt 更短，职责更清楚，工具也更少，稳定性比一个大一统 Agent 更高。最后由一个总控把各个 Agent 的输出汇总，生成最终回答。

12.记忆架构怎么做

​ 记忆一般分短期记忆、长期记忆和工作记忆。

  • 短期记忆保存最近几轮对话，用来保证多轮连续性；
  • 长期记忆保存用户偏好、稳定事实、长期设定；
  • 工作记忆保存当前任务的中间结果、已调过的工具结果和待确认状态。

​ 实际系统里，不会把所有历史聊天原文一直带给模型，那样 token 会越来越大。通常做法是保留最近几轮原文，把更早的历史压缩成摘要，再根据当前 query 去召回相关长期记忆。这样既能保留上下文，又能控制成本。

13.RAG 效果怎么提升？如果出现幻觉怎么办？

RAG 的提升一般从四层做：文档质量、召回质量、排序质量、生成约束。

  • 文档先做清洗、去重、结构化切片，减少脏数据和重复数据；
  • 召回阶段通常会做 query rewrite、混合检索和 metadata 过滤；
  • 排序阶段加 rerank，把真正相关的 chunk 放到前面；
  • 生成阶段通过 prompt 约束模型必须基于证据回答。

幻觉通常有三种来源：没召回到、召回错了、模型不按证据答。

  • 没召回到就优化切片和召回；
  • 召回错了就优化重排和过滤；
  • 模型乱答就加明确限制和拒答机制。

线上更稳的方式是把召回分数或证据置信度作为门槛，达不到阈值就直接拒答，而不是硬生成。

14.单个对话消耗的 token 大概多少？

​ 没有固定值，要看系统 prompt 长度、历史轮数、检索上下文多少、工具描述多少、模型输出多长。普通单轮问答可能几百到一两千 token，带 RAG 后经常会上到两三千甚至五千以上。多轮对话如果带历史摘要、工具调用结果和多个检索片段，单次请求上万 token 也很常见。

​ Agent 场景的 token 消耗通常比普通聊天更高，因为还包含工具 schema、状态信息、记忆和中间结果。线上一般会记录 prompt token、completion token、总 token、平均耗时、不同场景 token 分布，再做成本分析。

15.单轮或多轮对话 token 比较大怎么办？

​ 核心思路是裁剪、压缩、分流。单轮 token 大，通常是检索片段太多、工具描述太长、prompt 太啰嗦，可以减少 topK、先 rerank 再截断、把工具描述改成结构化 schema。多轮 token 大，通常是历史消息累积造成的，需要保留最近几轮原文，把更早历史压成摘要，长期偏好抽成结构化记忆。

​ 另外还可以做模型分级，让 query 改写、分类、路由、摘要这类轻任务用小模型，大模型只处理最终生成。输出端也要限制 max_tokens，避免回答过长。

16.算法和Infra工作有什么不同，侧重点分别是什么

  1. 关注对象不同

    • 算法：关心“模型为什么这样”，核心是效果、泛化、数据

    • Infra：关心“系统为什么会挂”，核心是稳定、性能、成本

  2. 表现不同

    • 算法：loss不降，指标无法提升

    • infra：服务直接挂

  3. 确定性

    • 算法是黑盒调试
    • infra可以定位问题

17.介绍MHA、GQA和MLA

MHA (Multi-Head Attention) - 多头注意力

工作原理：

  • 将输入映射为H个头（Heads）。
  • 每个头都有自己独立的 查询矩阵 ( W^Q^)、键矩阵 ( W^K^ ) 和 值矩阵 ( W^V^ )。
  • 如果有 H 个头，就会生成 H 组独立的 Key 和 Value

GQA (Grouped-Query Attention) - 分组查询注意力

由 Google 在 2023 年提出，旨在解决 MHA 显存过大和 MQA（Multi-Query Attention，所有头共享一组 KV，效果下降明显）效果太差的问题

  • 将 H 个查询头（Query Heads）分成 G组。
  • 每组查询头共享同一组键（Key）和 值（Value）

MQA是只有一对Key和Value

MLA (Multi-Head Latent Attention) - 多头潜在注意力

即使是 GQA，当模型层数很深、序列极长（如 128k+）时，KV Cache 依然占据巨大显存，成为瓶颈。

工作原理：

  • 低秩压缩：不直接存储庞大的 Key 和 Value 矩阵。
  • 它将 Key 和 Value 投影到一个低维的潜在向量（Latent Vector）（实际上是压缩h，通过down_k和down_v）空间进行存储。
  • 在计算注意力时，再从这个低维潜在向量动态地解耦恢复出完整的 Key 和 Value。
  • 简单来说：它把 KV Cache 进行了“有损压缩”，只在计算时才“解压”。

18.reflection

​ Reflection 是一种通过让模型对自身输出进行评估并迭代改进的机制。通常包括初始生成、自我评估和基于反馈的修正三个步骤。

​ 它可以通过多轮推理提升模型在复杂任务上的表现，例如代码生成和数学推理。常见实现包括 self-reflection、Reflexion 以及结合 CoT 或 ReAct 的方法。

​ 不过它的代价是增加推理成本和延迟，同时存在 critique 质量不稳定的问题。

Prompt → 初始答案 → 自我评估 → 修正 → 最终答案

Requery和Reflection 一个是对外再拿数据，一个是对内自我思考

20.GRPO

GRPO 通过在同一个 prompt 下采样多个候选输出，并利用这些输出之间的相对 reward 来构造 advantage，从而替代传统 PPO 中依赖的 value model，实现更简化的策略优化流程。

  • 不需要 value model
  • 更稳定的相对优化（同组比较减少 reward scale 问题）
  • 更适合 LLM 多样性采样

21.常见的深度学习API

从API抽象层级出发，区分“计算表达”“模型定义”“训练编排”“推理部署”四个维度

1.底层计算与编译层
  • 当前主流是 PyTorch 2.x (torch.compile) 和 JAX，它们通过即时编译技术自动优化计算图，开发者无需手动管理图结构。
2.中层模型构建API
  • **PyTorch (nn.Module)**：凭借动态图机制和灵活性，占据学术界和前沿研究的主导地位。
  • **Keras (tf.keras)**：作为 TensorFlow 的高阶API，凭借简洁性在工业界快速原型开发中仍有一席之地。
3.高层训练与部署层（当前最活跃领域）
  • 训练编排：Hugging Face Transformers 和 Accelerate 已成为NLP及多模态领域的事实标准API，屏蔽了底层分布式训练的复杂性。

Accelerate自动处理复杂的分布式训练和混合精度训练的样板代码,让你专注于模型和训练逻辑本身

  • 推理服务：在生产环境中，vLLM 构成了关键的部署API层，负责将模型高效转化为服务。

22.RoBERTa

核心贡献：即通过“暴力美学”（更大的数据、更久的训练）和工程细节的微调，挖掘出了 BERT 架构的极限

  1. 移除NSP（Next Sentence Prediction）任务；
  2. 采用更大规模、更高质量的训练数据（160GB纯文本，远超BERT的16GB）；
  3. 使用更长的训练时间、更大的batch size（最大8K序列）、更多训练步数（300K步，BERT为125K）；
  4. 动态掩码（Dynamic Masking）：每次向模型输入前实时生成掩码，而非预生成固定掩码；
  5. 取消词频驱动的采样（即不再使用BERT中的“whole word masking”策略的特殊实现，但RoBERTa默认采用标准WordPiece掩码，并在部分实验中验证了whole word masking并非关键）；
  6. 学习率预热后线性衰减，而非BERT的warmup+cosine decay；
  7. 取消dropout（在部分大batch实验中发现其影响较小，实际代码中仍保留，但论文强调其非关键）

23.ROI和ROI2

投资回报率

  • 传统 ROI = 广告直接带来的 GMV / 广告费
  • ROI2 (全域 ROI) = 直播间/店铺总 GMV / (广告费 + 变动成本) 或 仅看 总GMV/广告费（视平台定义略有不同，但核心是看整体盘子）

ROI2 不再纠结于“这一块钱广告费换回了多少直接订单”，而是看“这一块钱广告费是否撬动了整个直播间的生意爆发（包括自然流量带来的订单）

增量推广

通过科学实验设计（如A/B测试、随机对照试验RCT、差分法DID等），精准识别并仅针对“因本次推广行为而产生额外转化”的用户群体所开展的营销活动。

其核心目标是度量和最大化“真正由推广带来的净增量效果”，剔除自然转化、品牌惯性、季节性波动等混杂因素。

典型场景包括：向一组随机抽样的用户投放广告（实验组），另一组不投放（对照组），比较两组转化率之差，该差值即为增量转化率；再结合成本计算增量ROI（Incremental ROI = 增量毛利 / 推广投入）。

24.PLE

PLE 渐进式分层提取（Progressive Layered Extraction）是一种专为多任务学习（Multi-Task Learning, MTL）设计的共享-特化结构建模范式，

核心思想是通过层级化、渐进式解耦任务专属参数与跨任务共享参数，有效缓解多任务间的负迁移（negative transfer）和跷跷板现象（seesaw effect）

核心是 CGC（Customized Gate Control **定制化门控机制）结构，通过分离专家和共享专家来缓解冲突。**

通过多层的网络结构，逐步地从共享的专家网络中提取通用特征，并同时从特定的专家网络中提取任务独有的特征，从而实现多任务之间的平衡与优化

  1. PLE本身已在阿里系推荐系统中大规模落地；
  2. 后续提出的MMoE+PLE hybrid架构（如阿里2022年提出的PLE++）；
  3. 腾讯2023年提出的DSelect-K（Dynamic Selective Expert Routing），它将专家选择从静态门控升级为任务自适应top-k稀疏路由，并引入梯度感知的专家负载均衡约束；
  4. Google 2024年提出的TAPAS（Task-Aware Parameter Allocation Strategy），采用元学习框架联合优化参数分配策略与任务权重，在Criteo-MTL benchmark上相对PLE提升+3.7% AUC平均增益。

与MMoE的差别

MMoE（多门控混合专家 Multi-gate Mixture-of-Experts）和PLE（Progressive Layered Extraction）都是多任务学习（MTL）中用于建模任务间共享与特异性关系的深度神经网络结构，

核心区别在于：MMoE通过共享多个专家子网络 + 为每个任务独立设计门控网络（multi-gate），实现轻量级任务相关性建模，但未显式解耦共享与任务专属知识；

而PLE在MMoE基础上引入“共享专家+任务专属专家”的分层双路径结构，并采用渐进式分层提取机制（progressive layering）与门控路由的联合优化，显式分离共享知识与任务私有知识，显著缓解负迁移（negative transfer）并提升多任务性能。

25.指标对比baseline提升主要来自于什么

结合实际项目回答

指标对比baseline提升主要来自于模型结构优化、特征工程增强、训练策略改进（如学习率调度、正则化、数据增强）、高质量标注数据引入、损失函数设计更贴合业务目标，以及后处理策略的精细化调整。其中最核心且可复现的贡献通常来自对问题本质的深入理解所驱动的针对性改进，而非单纯堆叠复杂模块。

26.模型拆分前后的结构变化

模型拆分（Model Partitioning）是指将一个大型深度学习模型按层、按模块或按计算图节点，逻辑或物理地切分为多个子模块，并部署到不同设备（如多个GPU、CPU与GPU混合、边缘设备与云端协同等）上执行。

  • 拆分前，模型是一个完整的、连续的计算图，所有参数和前向/反向计算都在单设备内存与计算单元中完成；

  • 拆分后，模型结构变为分布式计算图：各子模块间通过显式的通信边（如张量传输）连接，引入了设备间数据移动（inter-device tensor transfer）、跨设备梯度传递、以及同步/异步执行调度机制，整体呈现为有向无环图（DAG）中带设备标签的子图划分。

从四个维度对比分析：

（1）计算图拓扑结构（是否仍为单连通DAG？是否存在跨设备边？）；

（2）内存布局（参数/激活/梯度是否跨设备分布？是否需序列化/反序列化？）；

（3）执行流程（前向是否需等待远程输入？反向是否需跨设备梯度回传与规约？）；

（4）系统接口（是否引入新的通信原语，如send/recv、all-gather、p2p all-reduce？）。

结构变化本质是将“单机单图”映射为“多机多子图+通信超边”的异构执行图。

ZeRO-1/2/3分别对优化器状态、梯度、模型参数进行分区存储

27.Uplift模型

Uplift模型（也称因果推断模型、增量建模模型或Treatment Effect模型）不是单一固定结构的模型，而是一类专门用于估计“干预（treatment）”对个体产生的净因果效应（uplift） 的机器学习模型。

其核心输出是每个样本的条件平均处理效应（Conditional Average Treatment Effect, CATE），即：τ(x) = E[Y(1) − Y(0) | X = x]，其中Y(1)为接受干预（如收到优惠券、看到广告、接受电话营销）下的潜在结果，

Y(0)为未接受干预下的潜在结果，X为协变量（用户特征）。输入是特征向量X（可能含干预标识T），输出是一个标量预测值——即该用户因干预而产生的额外响应提升量（例如：购买概率提升0.12，或LTV增加86元）。

28.AUC

随机选一个正样本 + 一个负样本，模型把正样本排在前面的概率

数值上：ROC 曲线下面积，ROC 曲线画的是：

  • 横轴：FPR（假正率）
  • 纵轴：TPR（真正率）
  • FPR和TPR根据分类的阈值变化而变化

计算

基于排序对（统计学本质）

$$
\mathrm{AUC}
= \frac{1}{P \cdot N}
\sum_{i=1}^{P} \sum_{j=1}^{N}
\mathbf{1}\big( f(x_i^+) > f(x_j^-) \big)
$$
P为正样本数量，N为负样本数量，其中简化为：
$$
\mathrm{AUC}
= \mathbb{P}\big( f(x^+) > f(x^-) \big)
$$

工程常见累加式

$$
\mathrm{AUC}
= \sum_{k=1}^{M}
TPR_k \cdot (FPR_k - FPR_{k-1}),
\quad
FPR_0 = 0,; TPR_0 = 0
$$

AUUC

因果推断 / uplift modeling（营销/推荐常用)。核心关注：推了有没有用

Uplift=P(Y∣T=1)−P(Y∣T=0)。uplift 曲线下面积:对累计曲线积分得到。

GAUC

每个用户各自算排序能力，再做平均

按用户分组计算 AUC，再加权平均

$$GAUC = \sum_{u} w_u \cdot AUC_u$$

29.一共64匹马，8个跑道，如何找到前四

首先分八组得到小组内的排名（8场）

然后八组的第一名进行排名（1场），此时获得A1为第一名。

然后根据第一名的结果，从小组中筛选出A2,A3,A4,B1,B2,B3,C1,C2,D1，将A4排除后，看最后结果，得到前四。（1场）

最坏情况：若最后的结果是A2,A3,B1那么有可能A4仍要和B1再比一场

30.反向传播过程池化层参数如何更新

反向传播的目的是计算损失函数对网络中各层参数的梯度，并利用这些梯度通过优化算法（如SGD、Adam等）更新参数。

然而，池化层（如最大池化、平均池化）并不包含任何需要学习的权重或偏置参数。

它的作用是下采样，减少特征图的空间尺寸，从而降低计算量并增强平移不变性。由于没有参数，自然也就不涉及参数更新。

31. L1、L2范式

  • L1：让参数变稀疏（很多变成0）–>更干净

    • 文本分类（上万维特征，“没用的特征直接干掉”）
    • 生物信息（基因数据）

  • L2：让参数变小（但很少变成0）–>更稳定

    • 回归问题
    • 大多数深度学习默认用 L2

L1 范式（L1 Regularization）

$$
|w|1 = \sum{i=1}^{n} |w_i|
$$

L2 范式（L2 Regularization）–最常用

$$
|w|2^2 = \sum{i=1}^{n} w_i^2
$$

加入到损失函数中一般是前面加上lambda调节

把 L1 / L2 加进损失函数，本质是在“惩罚复杂模型”，防止过拟合

33.新模型灰度测试指标不稳定，这时候负向如何理解

灰度测试指标不稳定时的“负向”，本质是：新模型在真实流量分布下的“期望收益低于旧模型”，但必须先排除噪声与分布偏移。

  • 方差问题（统计噪声）–>用户少，行为随机性大
  • 分布偏移–>早期用户更活跃
  • 冷启动–>用户行为还没适应

通过显著性、多指标一致性、分桶分析（重要）等方案判断是否需要回滚

37.这么多强化学习的算法，有什么心得？强化学习是怎么从最开始演变到现在的

本质：在不确定环境里，通过试错学会做长期最优决策

理论时代–>经典强化学习算法（Actor-Critic，策略梯度）–>深度学习网络首次应用（Q-learning）–>Transformer+RLHF+世界模型为代表的“认知增强型RL“

On-policy（同策略）和off-policy（异策略）

On-policy（同策略）和off-policy（异策略）是强化学习中按策略与数据采集策略是否一致划分的两大范式。

核心区别在于：

  • on-policy算法要求用于更新策略的样本必须由当前待优化的策略πθ（即最新参数下的策略）实时生成；
  • off-policy算法允许复用历史数据（甚至由完全不同的行为策略β生成的轨迹），只要满足重要性采样或Q函数解耦等条件即可。

GRPO（Generalized Reward-Predictive Policy Optimization）本质上是on-policy的，它继承自PPO框架，依赖当前策略rollout生成的完整轨迹进行优势估计与策略更新，不引入离线缓冲区或跨策略重要性权重修正。

真正的on-policy实现需同时满足三个条件：

（1）每次策略更新前必须用当前策略π_θ重新采集一批新轨迹（不能复用旧轨迹）；

（2）价值函数（如V或Q）的训练也仅使用这批新轨迹中的状态-动作对；

（3）所有梯度计算（包括策略梯度、优势估计、KL约束等）均基于该批次数据，且禁止任何形式的回放（replay）、优先采样（prioritized sampling）或跨轮次数据混合。

on-policy：必须用“当前策略”生成的数据训练

off-policy：可以用“别的策略”生成的数据训练

39.OpenClaw

核心思想是通过统一的 Gateway 进程管理所有外部通讯平台的链接，通过AgentRuntime进行各个大模型的后台调用，所有产生的数据、记忆全部存储在本地磁盘上

记忆系统

OpenClaw 的记忆系统采用了一个双源记忆架构的设计，其将记忆整体分为了两类：每日日志（动态记忆）和长期记忆（静态记忆）。

记忆类型 存储格式 存储路径 产生方式
静态记忆 Markdown 文件 ~/.openclaw/workspace/MEMORY.md 和 memory/*.md 手动创建 + 自动生成
动态记忆 JSONL 文件 ~/.openclaw/agents/{agentId}/sessions/*.jsonl 自动记录

OpenClaw 所代表的应用，其对“记忆（Memory）”的定义是：持久存储在磁盘上的结构化信息。

因此，只要有足够的存储空间，记忆就可以无限增长，能够跨会话保留，在使用时按需检索，且存储成本几乎为零。更为关键的是，这种记忆能够支持语音搜索，也就意味着不需要把所有历史信息都塞进上下文，只需要检索当前任务相关的片段。
